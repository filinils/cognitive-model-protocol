# Why CMP Exists

Large language models are increasingly used as collaborators rather than one-off tools. They help people write, program, reason, plan, remember, research, negotiate uncertainty, and refine ideas over time.

Long-term collaboration requires some form of working model. An AI system that adapts to a person across sessions is already making assumptions about what matters, what the person tends to prefer, where they need help, what kinds of feedback are useful, and how the collaboration might proceed. Even when no explicit profile is exposed, the system may still operate from implicit patterns inferred through interaction.

CMP begins from a simple claim: if working models are being used, they should be explicit enough to inspect, challenge, revise, and evaluate.

## The Risk of Implicit Models

Implicit models create practical and ethical risks.

Opacity makes it difficult for a user to know what an AI system believes about the collaboration. The system may act from hidden assumptions that the user cannot see or correct.

Misclassification becomes more likely when observed behavior is compressed into labels too early. A person may be treated as a type rather than as a collaborator in a particular context.

Paternalism can emerge when a system silently decides what is best for the user. Even helpful adaptation can become constraining if the user cannot inspect or contest the basis for it.

Extraction becomes easier when behavioral inferences are treated as platform assets rather than as shared working material. A model that exists only inside one implementation can quietly become a form of lock-in.

Scientific comparison becomes harder when every system represents its assumptions differently. If models cannot be inspected or compared, it is difficult to evaluate whether they improve collaboration or simply make it feel smoother.

## The Opportunity of Explicit Models

Explicit models create a different possibility.

Transparency allows a user to see the current working hypotheses that shape collaboration.

Correction allows the user to challenge a claim, add missing context, or reject a pattern that no longer applies.

Portability allows a collaboration model to move across tools without requiring the user to start from nothing or surrender their context to one platform.

Evaluation becomes more concrete because hypotheses can be tested against evidence, revised over time, and compared across implementations.

Better collaboration becomes possible when the model is treated as shared working material rather than as an invisible classifier.

## What CMP Is Trying to Represent

CMP does not try to define who a person is. It does not claim to solve identity, consciousness, personality, diagnosis, or moral status.

CMP focuses on practical collaboration between humans and AI systems. The goal is to represent revisable hypotheses about how collaboration may work better in a given context.

Those hypotheses may concern interaction patterns, useful forms of feedback, recurring uncertainty, matching signals, complementary collaborator roles, or conditions under which the model should be reconsidered. Each element should remain tied to evidence, confidence, uncertainty, and possible falsifiers.

## The Central Shift

The central object in CMP is a `collaboration_model`, not a `user_profile`.

That distinction matters. A user profile suggests a relatively stable description of a person. A collaboration model suggests a working representation of a relationship between a person, an AI system, a context, and a purpose.

The model should remain provisional. It should be open to challenge. It should preserve uncertainty. It should make room for the user to say: that was true once, that is only true in this domain, that is an overreach, or that hypothesis is useful but incomplete.

## The Aim

CMP aims to make human-AI working models more inspectable, evidence-based, revisable, portable, and falsifiable.

It is a draft, not a standard. Its immediate purpose is to invite criticism, alternative formulations, examples, ethical objections, and concrete implementation feedback.
